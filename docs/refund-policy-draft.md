# PROJECT: HW 환불 정책 초안

## 목적

이 문서는 운영/CS/개발 기준을 맞추기 위한 환불 정책 초안이다. 실제 출시 전에는 결제 제공자 정책, 서비스 약관, 관련 법무 검토를 거쳐 확정해야 한다.

## 기본 원칙

1. 환불 가능 여부는 서버 구매 기록과 지급 기록을 기준으로 판정한다.
2. 미사용 유료 블링은 환불 가능 상태로 본다.
3. 이미 소비된 유료 블링은 기본적으로 환불 불가 상태로 본다.
4. 직접 구매 상품은 미사용 또는 미수령 상태일 때 환불 가능 상태로 본다.
5. 환불 승인 시 지급된 항목은 회수하거나 권한을 비활성화한다.
6. 환불 처리 결과는 구매 기록, 지급 기록, 장부 기록에 남긴다.

## 상태별 기준

| 대상 | 상태 | 초안 처리 |
| --- | --- | --- |
| 유료 블링 | 전액 미사용 | 환불 가능 |
| 유료 블링 | 일부 사용 | 사용분 제외 또는 제한 검토 |
| 유료 블링 | 전액 사용 | 환불 불가 원칙 |
| 직접 현금 상품 | 미지급 | 환불 가능 |
| 직접 현금 상품 | 지급됐지만 미사용 | 회수 가능하면 환불 가능 |
| 직접 현금 상품 | 사용됨 | 환불 불가 원칙 |
| 영구 권한 | 미사용/회수 가능 | 회수 후 환불 가능 |
| 운영 지급 | 무료 지급 | 현금 환불 대상 아님 |

## 환불 처리 흐름

1. CS 또는 사용자가 환불 요청을 생성한다.
2. 서버가 `purchaseId` 기준으로 구매 상태를 조회한다.
3. 서버가 지급 기록과 장부 기록을 조회한다.
4. 서버가 회수 가능 여부를 계산한다.
5. 운영자가 승인/거절한다.
6. 승인 시 서버가 지급 항목 회수 또는 권한 비활성화를 수행한다.
7. 서버가 PG 환불 요청을 처리한다.
8. 서버가 `RefundCase`, `Purchase`, `GrantRecord`, `CurrencyLedger` 상태를 갱신한다.

## 유료 블링 환불 계산

```ts
type RefundablePaidBling = {
  purchaseId: string;
  grantedPaidBling: number;
  spentPaidBlingFromPurchase: number;
  remainingRefundablePaidBling: number;
};
```

초기 구현에서는 구매 단위별 유료 블링 사용 추적을 단순화하기 위해 `currency_ledger.sourceId`와 소비 트랜잭션 ID를 유지한다.

## 환불 상태

```ts
type RefundStatus =
  | "requested"
  | "reviewing"
  | "approved"
  | "rejected"
  | "completed";
```

## 완료 기준

운영자가 특정 `purchaseId`를 보고 환불 가능 여부, 회수 필요 여부, 환불 후 상태 변경을 판단할 수 있어야 한다.
