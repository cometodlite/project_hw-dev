# PROJECT: HW 결제 선행 전제

## 목적

이 문서는 PROJECT: HW에 결제 시스템을 붙이기 전에 고정해야 하는 계정, 저장, 재화, 상품, 환불, 복원 기준을 정리한다. 현재 문서는 구현 문서가 아니라 설계 기준 문서이며, 실제 PG 연동과 법무 검토 전까지는 초안으로 유지한다.

## 확정 전제

| 항목 | 기준 |
| --- | --- |
| 서비스 구조 | 계정 기반 웹게임 |
| 결제 채널 | 웹 전용 |
| 게임 방향 | 힐링 생활형 RPG, 생활/하우징/시간 연출 중심 |
| 블링 구조 | 내부적으로 `freeBling` / `paidBling` 분리 |
| 판매 방식 | 블링 충전 상품 + 직접 현금 상품 판매 병행 |
| 저장 구조 | 서버 권위 저장 + 로컬 편의 저장의 하이브리드 |
| 환불 기준 | 미사용 유료 재화/상품 기준 우선 |
| 복원 기준 | 동일 계정 로그인 시 서버 기록 기준 자동 복원 |

## 핵심 원칙

1. 결제는 익명 사용자에게 제공하지 않는다.
2. 구매, 지급, 회수, 환불과 연결되는 데이터는 서버를 기준으로 판정한다.
3. `paidBling`과 `freeBling`은 UI에서 합산 표시할 수 있지만 서버 저장과 장부에서는 분리한다.
4. 모든 재화 변화는 `currency_ledger`에 기록한다.
5. 모든 결제 요청과 지급 요청은 멱등 키를 가진다.
6. 클라이언트의 로컬 저장은 편의 기능이며 결제 상태의 근거로 사용하지 않는다.
7. 상품 지급은 구매 기록과 지급 기록을 모두 남긴 뒤 완료 처리한다.
8. 환불과 복원은 구매 기록, 지급 기록, 장부 기록을 함께 조회해 처리한다.

## 저장 구조 기준

서버 저장 대상은 계정, 코인, `freeBling`, `paidBling`, 인벤토리, 하우징, 해금 상태, 구매 내역, 지급 기록, 장부 기록이다. 로컬 저장 대상은 UI 설정, 필터, 연출 옵션, 마지막 선택 패널 같은 비핵심 편의 상태로 제한한다.

## 결제 흐름 기준

1. 클라이언트가 로그인 상태를 확인한다.
2. 서버가 상품과 가격을 조회한다.
3. 서버가 결제 요청을 생성한다.
4. PG 또는 결제 제공자의 성공 콜백을 서버가 수신한다.
5. 서버가 결제 결과를 검증한다.
6. 서버가 구매 기록을 저장한다.
7. 서버가 상품 지급을 처리한다.
8. 서버가 지급 기록과 장부 기록을 저장한다.
9. 클라이언트가 서버 상태를 다시 동기화한다.

## 직접 현금 상품 기준

직접 현금 상품은 블링 충전 없이 바로 지급되는 상품이다. 예시는 가구 패키지, 시즌 패키지, 후원 패키지다. 직접 현금 상품도 `purchases`, `entitlements`, `grant_records`, `currency_ledger` 중 필요한 기록을 남긴다.

## 환불/복원 기준

환불은 미사용 기준을 우선한다. 유료 블링은 사용 전 잔액이 남아 있을 때 환불 가능 상태로 판정한다. 직접 구매 상품은 미사용 또는 미수령 상태를 환불 가능 기준으로 삼는다. 영구 상품과 서버 잔액은 동일 계정 로그인 시 자동 복원한다.
