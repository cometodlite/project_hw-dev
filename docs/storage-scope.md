# PROJECT: HW 저장 범위 명세

## 목적

현재 PROJECT: HW는 브라우저 로컬 저장 중심의 프로토타입이다. 결제 도입 이후에는 결제와 연결되는 핵심 데이터가 로컬 저장에 의존하면 안 된다. 이 문서는 서버 저장과 로컬 저장의 경계를 고정한다.

## 서버 권위 저장 대상

| 데이터 | 이유 |
| --- | --- |
| `coin` | 경제 밸런스와 판매 결과 추적 |
| `freeBling` | 무료 지급 재화 추적 |
| `paidBling` | 유료 재화, 환불, 복원 추적 |
| `inventory` | 구매 상품, 지급 상품, 회수 대상 포함 |
| `housing` | 구매 가구와 배치 상태 복원 |
| `unlocks` | 상품/숙련도 해금 상태 유지 |
| `purchases` | 누가 무엇을 샀는지 기록 |
| `grantRecords` | 무엇이 지급됐는지 기록 |
| `entitlements` | 영구 상품/권한 보유 상태 |
| `currencyLedger` | 모든 재화 변동 장부 |
| `refundCases` | 환불 요청과 처리 상태 |
| `restoreEvents` | 복원/동기화 처리 기록 |

## 로컬 저장 대상

| 데이터 | 예시 |
| --- | --- |
| UI 설정 | BGM 켜기, 패널 접힘 여부 |
| 편의 상태 | 마지막 선택 패널, 정렬/필터 |
| 연출 옵션 | 저사양 모드, 애니메이션 감소 |
| 임시 캐시 | 최근 표시용 상품 목록 캐시 |

로컬 저장 데이터는 언제든 삭제되어도 서버 상태를 복구하는 데 문제가 없어야 한다.

## 하이브리드 저장 규칙

1. 클라이언트는 로그인 후 서버 저장 데이터를 먼저 불러온다.
2. 로컬 저장 데이터는 UI 편의 상태에만 병합한다.
3. 결제와 연결되는 데이터는 클라이언트에서 직접 확정하지 않는다.
4. 오프라인 상태의 플레이는 결제 연동 데이터에 반영하지 않는다.
5. 서버 동기화 실패 시 결제, 지급, 회수 버튼은 제한한다.

## 저장 버전 필드

서버 저장 데이터에는 버전과 마이그레이션 필드를 둔다.

```ts
type PlayerSave = {
  userId: string;
  saveVersion: number;
  schemaVersion: "2026-04-16";
  coin: number;
  wallet: {
    freeBling: number;
    paidBling: number;
  };
  inventory: Record<string, number>;
  housing: {
    slots: Array<string | null>;
  };
  unlocks: Record<string, boolean>;
  updatedAt: string;
};
```

## 동기화 우선순위

| 상황 | 처리 |
| --- | --- |
| 첫 로그인 | 서버 기본 저장 데이터 생성 |
| 기존 로컬 저장만 있음 | 서버 계정 연결 전까지 결제 비활성 |
| 서버/로컬 충돌 | 서버 값을 우선하고 로컬 UI 설정만 병합 |
| 결제 성공 직후 | 서버 지급 완료 후 클라이언트 재동기화 |
| 환불/회수 후 | 서버 상태를 다시 받아 클라이언트 표시 갱신 |

## 완료 기준

결제, 지급, 환불, 복원에 필요한 데이터가 모두 서버 기준으로 정의되어 있고, 로컬 저장은 UI 편의 상태로만 제한된다.
